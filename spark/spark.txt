Spark:
	
	- Spark is a open source application, 2009 released.
	- Apache spark is a unified analytics engine for large-scale data processing. It support Java, Python, R, Scala.
	- Four Core Modules:
		- Spark SQL, Dataframe
		- Spark Structured steaming.
		- Spark Mlib
		- Spark Xgraph
		
	Spark workflow:
	
		- submit the spark application
		- Spark context:
			Dag scheduler:
				- unsolved query plan -> optimized logical plan -> physical plan 
				
			Task scheduler:
				- task 
		- Resource negotiation
		- Instruction to available worker.
		- Executor registors with driver 
		- Driver monitors and palns for future task.
		
			
	
	- Modes of Deployment:
		=> the deployment mode specifies where the driver program will be run.
		
		- Cluster mode:
			- driver program and executor runs on the same machine. [ YARN, Mesos, Kubernetes ]  
			- Its is best for production.
			- Spark driver component does not run on the machine from which the spark job has been submitted.
			- Driver may be running as one of the worker nodes.
			
		- Client mode:
			- driver program run in local and the executor runs on the cluster node.
			- it is best for development phase.
			- Spark driver component runs on the machine node from where the spark jobs is submitted.
			- Driver is launched within the application.
			
		- Standalone:
			- Self managing runs workloads. No need for external cluster manager.
			- Its best for testing purpose.
			- Java must be installed on each machine.

		- Faster Processing tool compared to hadoop's Mapreduce.

		- Lazy Execution. Program start Transformations after action.
		- In memory Computation. (Stored in RAM)

	Architecture:
		- Master & Slave Architecture.
		- Master:
			- Driver Programe(Spark session, spark context)
			- Every application starts with spark context.
			- It communicate with cluster Manager and Slave Nodes.
			- The driver manager receive the code from user and It told the cluster manager to allocate the resource as per the user defined.
			- The Driver Programe pass the code to Worker node for the execution. The the result will pass to the Driver Programe.
			- It is responsible for coordinating and distributing the work across multiple nodes.
			- It schedule the task, track the state of worker node.
			
		- Worker Node:
			- It runs the application in cluster and reports to driver program.
			- Jobs: process data, store the data, handle the network.
			- Each node having its own executors.
	
	DAG in spark:
		- Directed Acyclic Graph with no directed cycles. It is sequential manner of execution (transformation of RDD).
		- It helped for data lineage.
		- through lineage it achieve falut tolerance and resilient.
		
	Apache streaming:
		- SOurce: Kafka, etc.
		- Get the data and move them to spark processing (it is as batch process some x sec)
		- It following round robin method.
		- Types:
			- Reliable: Receiver send acknowledgement to the source post success. It replication on the spark storage space
			- Unrelaiable: There is no acknowledgement sent to the data sources.
		
	Replication and Coalesce:
		Replication:
			- Increase or decrease the data partition.
			- repartion create a new data partition and perform full shuffle of evenly distributed data.
			- Its internally calls coalesce. Thereby making it slower then coalesce.
		Coalesce:
			- Readuce partion only.
			- reduce amount of shuffled data unevenly.
			- if unequal-sized partitions, speed might be slightly slower.
		
		Shared variables or Miminized data transfer:
			Broadcast variables :
				- allow programmer to keep a read only varibale cached on each machine rather than shipping a copy of it with tasks. Limit 10MB.
			
				- The result is stored in local for quick access.
				
				- it is read only. cached in memory, reduce network traffic.
				
				- It increases the efficiency of the join between large and small RDD.
				
				- A broadcast variable is entirely cached on each worker node so it doesn't need to be shipped or shuffled between nodes with each stage
			
			Accumulator:
				- It is used for aggreagating information across the executors.
				
				- Updated only by worker tasks using += operator.
				
				- read only by driver program (spark context, sparksession).
				
				- Use case: no.of errors, invalid record, gathering info for debugging or logging
				
				- sc.accumulator(0).value
				
				- Update the variable value in parallel during execution.
				
	Shuffling:
		- Process of redistribution of the data across different partition.
		
		- which might or might not cause data movement across the JVM processes or the exucutors on the seperate machines is known as shuffling or repartition. 
		
		- Parition is nothing but a smaller logical division of data.
		
		- groupByKey() , reducebyKey() , join() , groupBy() e.t.c.
		
	YARN in spark:
		- Yet Another Resource Negotiator.YARN is a cluster management technology.
		
		- Managing and allocating resources such as CPU, memory, and storage
		
		- Monitoring and managing workloads.
		
		- app submit -> resource manager allocate resource for running an application -> tell node to create the container to run the task. -> the task executed in the container.

	Executor Memory:
		- The amount of memory allocate to each executor process to store intermidiate data, cached data and execution buffers in a spark application.
		
		- Executors are responsible for performing data processing and computation tasks.
		
		- Formula for memory overhead:
			-  max( executor memory * 0.1, 384mb)

	Demerits of spark:
		- It may lead to some memory based problem due its use of more storage spark.
		
		- Cost efficient in bigdata processing due to is in-memory computation.
		
		- * when using large number of small files (gzipped). spark needs to uncompress these files by keeping them on its memory and network.

		- So, large amount of time is spent in burning core capacities for unzipping the files. 
		
		- spark doesn't work well in multi-user environments as it is not capable of handling many users concurrently.

	Steps to calculate executor memory:
		
		- Number of nodes/machine = 10
		- Number of cores in each node = 15 cores
		- RAM of ech node = 61 GB
		
		Core:
			- no.of cores = no.of concurrent tasks that can be run parallelly by the executor . the optimal value as part of the general rule of thumb is 5
			
		Executor:
			
			No.of executors for single node = No.of cores / concurrent task
			
			 = 15 / 5
			 = 3
			 
			 
			 No.of executor for total = No.of nodes * Number of executors in each node
			 
			 = 10 * 3
			 = 30 executors per spark job.
