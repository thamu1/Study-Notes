	Pods:
	------
		
		Meaning:
			- Deployable atomic unit is Pod in kube. but in docker deployable atomic unit is containers.
			- Pod is a wrapper that has one or more containers.
			- containers can runs an application inside a top-level pod even if the pod contains single container.
			
			1. Package app as image and run as container.
			2. Wrap containers in a POD.
			3. Use manifest file(YML) to deploy or create pod(s). using deployment yaml can create multiple pods.
		
		Life cycle:
			1. Pending state: Pod has need accepted by the API, but not serviced yet.[request received searching for node]
			2. Creating state: scheduler identified nodes/vm and pulls relevant images. [start creating the env]
			3. Running state: Tries to start the containers. It will retry again and again [run the container]
			4. CreashBackoff: After tries so many times to run the container if it is not run then it will move to crashBackoff.
			
			*. Stuck in pending: If node not identified. its called pending state.
			*. stuck in creating: If image not found.
			
			#. init containers: 
				- It is a specialized containers with start-up code.
				- Runs before app containers in POD.
				
		Summary:
			- PODs are automic units.
			- container wrapper.
			- pod create using kind (pod/deployment) in Yaml file.

		YAML with multiple containers [Deployment]:
		-----------------------------------------
		
			- Kubernetes application configured using YAML files. this will allow us to control Kubernetes app with ease.
			- It will allow JSON or YAML.
			- It's key-value pair, lists, maps, list of maps and map of lists.
			- four fields must need => apiVersion, kind, metadata, spec.
			
			apiVersion:
				- it clear that it is related to API and it used to create the kubernetes object.
				- value will be v1 or app/v1
				
				ex: apiVersion: v1
			
			kind:
				- It define the type of object we aim to create. It will linked to apiVersion.
				- Objects [Pod, Depolyment, Service, Job or DemonSets]
				
				ex: kind: Pod
			
			metadata:
				- metadata provides unique properties of that object.
				- Have [name, namespace, label]
				
				ex: 
				metadata:
					name: nginx
					
			spec:
				- In spec we define the operation(work) of the object.
				- Have [containers, memory requirement, storage volume, network, other details]
				- [note] make sure to properly indent.
				
			ex:
			spec:
				containers:
				- name: nginx
				  image: neginx: 1.14.2
				  ports:
				  - containerPort: 80
			
						
			
			nginx deployment:
			-----------------
			
			apiVersion: v1
			kind: Deployment
			metadata:
				name: nginx-deployment
				labels:
					app: neginx-app
			spec:
				replicas: 3
				selector:
					matchLabels:
						env: prod-pod
				template:
					metadata:
						name: nginx-pod
						labels:
							env: prod-pod
							type: front-end
							identifier: nginx-fe-prod-pod
					spec:
						containers:
							- name: nginx-con
							  image: nginx:Latest
							  ports:
								- containerPort: 80
							
						
			Summary:
				- POD yaml = apiVersion, kind, metadata, spec
				- Deployment yaml = apiVersion, kind, metadata, spec [replica, selector, template(container details -> metadata, spec)]

	Namespace:
	----------
	
		- Namespace are virtual clusters backed by the same physical cluster. [it helps to create a virtual cluster in physical cluster]
		- It mainly used for segregation.
		
		Key points:
			- Namespace should be unique and resources within the namespace should also be unique.
			- same resource name can exist in different Namespace.
			- avoid create namespace with KUBE- / kube-.
			- Helps to group the resources. thats why it is creating the virtual cluster in physical cluster.
			
			
		Create Namespace:
			- connect with kubernetes.
			- kubectl get namespaces
			- kubectl create namespace <ns_name>
			- we can create this namespace using spec (.yml) - apiVersion, kind: Namespace, metadata, spec
			- kubectl apply -f myns.yml
			- kubectl get pods -n hr-ns
			- kubectl describe pod <pod name or id>
			- kubectl describe pod <pod name or id> -n <namespace_name>
			
	Pods:
		- Pod is a collection of one or more containers that operate together. 
		- Pods are the smallest unit of execution in Kubernetes.
	
	Nodes:
		- Nodes is a machine, having necessary services to run pods including CPU and memory resources they need to run.
		- Machine that runs containerized workload as part of a Kubernetes cluster.
		- Node can be Physical or Virtual machines and can be hosted on-premises or in the cloud.
	
	Control Plane:
		- Nodes managed by control plane.
		- This automatically handles scheduling the pods across the nodes in the cluster.
		
	Deamon set:
		- Everytime new node added to a cluster, the pod is added to it and when a node is removed from the cluster the pod is removed.
		- It used to ensure that all eligible nodes run a copy of a pod.
		- DeamonSet controller creates a Pod for each eligible node and spec.
		- It manages the number of pod copies run in a node.
		- Single copy of your app must run on all or subset of the nodes in the cluster.
