Query optimization :

1. View the query execution flow:
	`{PROJECT_ID}`.`region-{REGION_NAME}`.INFORMATION_SCHEMA.JOBS
	
2. Use the data preview options, Query specific columns, Use partitioned tables, Use SELECT * EXCEPT

3. bigquery-public-data.noaa_gsod.gsod194* = queries all tables from the 1940s.

4. Avoid tables sharded by date, Avoid oversharding tables

5. Prune partitioned queries, Use INT64 data types in joins instead of STRING data types to reduce cost and improve comparison performance.

6. Reduce data before using a JOIN, one complex query as multiple smaller and simpler queries instead.

7.  Use a WHERE clause to limit the amount of data a query returns. When possible, use BOOL, INT, FLOAT, or DATE columns in the WHERE clause.

8. If you are using SQL to perform ETL operations, then avoid situations where you are repeatedly transforming the same data.

9. Use procedural language (like stroed proc) , variables, temporary tables, and automatically expiring tables to persist calculations and use them later in the query.
	ex:
		SET (corpus_count, word_count) = ( SELECT AS STRUCT COUNT(DISTINCT corpus), SUM(word_count) FROM bigquery-public-data.samples.shakespeare
		WHERE LOWER(word) = target_word);

10. Join:
	- Avoid repeatedly joining the same tables and using the same subqueries.
	- For queries that join data from multiple tables, optimize your join patterns by starting with the largest table.
	

11. ORDER BY clause:
	- Placing an ORDER BY clause in the middle of a query, unless it is being used in a window function.
	- regular expressions and mathematical functions, to the end of the query.
	- If you are ordering a very large number of values, use a window function, and limit data before calling the window function. 
		SELECT id, reputation, creation_date,
		DENSE_RANK() OVER (ORDER BY creation_date) AS user_rank
		FROM bigquery-public-data.stackoverflow.users ORDER BY user_rank ASC LIMIT 10;
		
12. avoiding query anti-patterns:
	- Instead of using self-joins, use a window (analytic) function. [result of using a self-join will make square the time of result.]
	-  Avoid DML statements that update or insert single rows
	- Use  APPROX_TOP_COUNT to determine if the data is skewed [ex: Max, AVG different time complexity]. Filter your data as early as possible.
	- Unbalanced joins:
		- Pre-filter rows from the table with the unbalanced key.
		- If possible, split the query into two queries.
13.  Use column and table aliases when you work with similarly named columns across queries, including subqueries.
