			- same configration for all(create from identical template)

			- allow to have different configuration VM instance in the same group.
			- Features: Not auto scaling, healing, other services.

		Services and ingress:
			- Services are sets of pods with a network endpoint that can be used for discovery and load balancing.
			- Ingress are collection of rules for routing external HTTP(s) traffic to services.
		
		Google kubernetes Engine cluster:
			Cluster:
				- group of compute engine instances
				- type:
					- Master node : manages the cluster
					- worker node : run your workloads.
				- Master node:
					- all kubctl command take and control everything.
					- API server: handles all communication for a k8s cluster(from node and outside)
					- Scheduler: Decides placement of pods.
					- Control Manager: Manges deployments and replica sets.
					- etcd: Distributed database storing the cluster state.
				- Worker node:
					- Run your pods.
					- kubelet: Manage communication with master node(s).
			
			Cluster Types:
				- Zonal cluster:
					- Single zone: single control plane. nodes rnning in the same zone.
					- multi Zone: single control plane but nodes running in multiple zones.
				- Regional cluster: 
					- replicas of the control plane runs in multiple zones of the given region. Nodes are run in same zones where control plane runs.
				- Private cluster:
					- VPC native cluster. Nodes only have internal IP addresses.
				- Alpha Cluster:
					- Cluster with alpha APIs. early feature API's. used to test new k8s features.
					
		Kubernates - Pods:
			- smallest deployable unit in kubernetes.
			- a pod contains one or more containers.
			- each pod is assigned an ephemeral IP address.
			- FLow: deployment -> replica set -> pod -> containers.
			- all containers in a pod sharing the following.
				- network, ip address, storage, Ports, volumes(shared persistent disks)
			- POD status: running, pending, succeed, filed, unknown.
		
		Deployment vs replica set:
			- Deployment:
				- its create for each microservices
				- kubectl create deployment m1 --image=m1:v1
				- deployment represents a microservices(with all its releases)
				- deplyment manages new relases ensuring zero downtime.
			- Replicasets:
				- ensures that a specific number of pods are running for a specific microservice version.
				- kubectl scale deployment m2 --replicas=1
				- even fi one of the pods is killed, replica set will launch a new one.
			- Deploy v2 of microservice create a new replica set
				- kubectl set image deployment m1=m1:v2
				- v2 replica set is created.
				- deployment updates v1 and v2 replica sets based on the release strategies. 
		
		Kubernetes services:
			- Each pod has its own IP address.
			- If pod filed and replaced by replica set.
			- all old releases replaced by one of new releases.
			- kubectl expose deployment name --type=LoadBalancer --port=80
			- Types:
				- clusterIP: 
					- expose service on a cluster internal ip.
					- (use case): you want your service only be availeble inside the cluster.
				- LoadBalancer:
					- exposes services externally using a cloud porviders load balancer.
					- (use case): you want to create individual load balancer for each microservice.
				- NodePort:
					- exposes service on each node's IP at a static port
					- (use case): don't want to create external load balancer. instead create one ingress component to load balance multiple microservices.
					kubernetes engine -> services & ingress -> create instance.
					
		- Ingress:
			- these are collections of rules for routing external HTTP(s) traffic to services. can create ingerss using service tab.	
			- we can redirect the traffic to different backent based on the path.
			- (recommanded): NodePort service for each microservice. expose using a ingress.
			- Advantage:
				- if create one ingress create One load balancer and can serve multiple microservices.
				 
		- Image repository:
			- If you create a image for your microservice then store it in.
				- Container registry: fully managed container registry provided by GCP.
				- Docker Hub.
				- can be integrated to CI/CD tools to publish images to registry.
			- you can secure your container images. Analyze for vulnerabilities and enforce deployment policies.
			- Naming: Hostname/ProjectIF/Image:Tag-gcr.io/projectname/{holloworld}:{1}

		defination:
			- Google Cloud Functions is a serverless execution environment for building and connecting cloud services. With Cloud Functions you write simple, single-purpose functions that are attached to events emitted from your cloud infrastructure and services. Your function is triggered when an event being watched is fired. Your code executes in a fully managed environment. There is no need to provision any infrastructure or worry about managing any servers
			
		Types of Cloud Functions:
			There are two types of Cloud Functions:

				- which handle HTTP requests and use HTTP triggers. See Write HTTP functions for information about implementing HTTP functions.

			Event-driven functions : 
				- which handle events from your cloud environment and use event triggers as described in Cloud Functions triggers. See Write event-driven functions for information about implementing event-driven functions.
